# TODOS

## TODO 1: 为解析失败邮件设计独立的解析重试能力

- What: 为 `parse_status=failed` 的原始邮件新增独立的“解析重试”能力，与本轮“提取失败重试”分开设计和交付。
- Why: 当前这轮只覆盖 `extraction_status=failed`，解析失败邮件仍然只能停留在错误态，用户容易误以为“重试功能已经全覆盖”。
- Pros: 将“解析失败”和“提取失败”两类状态机拆开，后续错误提示、权限控制、任务模型和测试边界都会更清晰。
- Cons: 需要额外定义解析失败的可重试条件、重试次数规则、是否允许复用原始内容，以及和现有提取链路的衔接方式。
- Context: `原始邮件提取失败重试` 这轮已经明确不把 `parse_status=failed` 纳入范围，避免在一个 PR 里同时扩展两套失败语义。后续如果不单独规划，很容易在页面和接口上把两个概念混在一起，导致状态判断和用户认知都变差。
- Depends on / blocked by: 依赖本轮异步重试任务模型先落地，包括 `TaskLog` 复用、任务状态查询、权限控制、去重策略和统一事务边界。
