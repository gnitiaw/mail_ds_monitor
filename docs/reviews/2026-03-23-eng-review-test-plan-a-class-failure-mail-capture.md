## Affected Pages/Routes
- `/login` — 最小登录是否正常，是否只允许试点用户进入
- `/failure-queue` — 失败邮件队列是否展示完整、状态是否正确
- `/failure-queue/:id` — 失败邮件详情、命中规则快照、状态操作
- `POST /api/v1/capture-runs/replay` — 手动补跑是否可触发且不重复入队
- `GET /api/v1/failure-queue` — 列表查询是否只返回试点邮箱范围内数据
- `PATCH /api/v1/failure-queue/:id/status` — `new -> acknowledged -> resolved` 状态流转

## Key Interactions to Verify
- operator 登录后可以查看失败队列
- operator 可以进入详情页并看到命中原因与关键提取字段
- operator 可以把 `new` 改为 `acknowledged`
- operator 可以把 `acknowledged` 改为 `resolved`
- supervisor 能收到短窗口聚合提醒邮件
- supervisor 能收到固定频率汇总邮件
- 手动补跑不会产生重复队列项

## Edge Cases
- 真实失败邮件命中规则但缺少任务标识，仍能入队
- 非目标邮件不会误入队
- 同一封邮件被自动轮询和手动补跑重复扫描，不会重复入队
- 某个邮箱拉取失败，不影响其他试点邮箱继续处理
- 提醒邮件发送失败，不影响失败邮件入队
- 未登录用户不能访问失败队列
- operator 不能看到试点范围外邮箱的数据

## Critical Paths
- 自动轮询读取试点邮箱 -> 规则命中 -> 入队成功 -> 监督人收到短窗口提醒
- operator 登录 -> 查看失败队列 -> 打开详情 -> acknowledged -> resolved
- 手动补跑指定邮箱 -> 已存在失败邮件不重复入队 -> 队列状态保持正确
- 固定频率汇总 -> 仅汇总未解决项 -> supervisor 收到邮件
