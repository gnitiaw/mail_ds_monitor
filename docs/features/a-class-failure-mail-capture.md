# 功能名称：A 类客户关键失败邮件捕获层试点

## 一、背景
当前售后服务部对 A 类客户的关键批量任务失败邮件，仍依赖“第一负责人自己盯邮件”的人工方式处理。邮件分散在不同人的收件箱中，没有统一收口，也没有复核和监督机制。一旦失败邮件被遗漏，通常不是内部先发现，而是客户先发现数据错误或延迟，然后由售后服务部再手工跑数据补救，修补成本高，也会影响客户满意度和下游业务。

现阶段不适合继续扩展为“大而全的邮件管理平台”。本次试点的目标是收敛到一个最小、可验证、可快速上线的失败邮件捕获层，先解决“关键失败邮件会不会被悄悄漏掉”这个核心问题。

## 二、目标
- 在明确的一组试点邮箱范围内，稳定捕获 A 类客户的关键批量任务失败邮件
- 将目标失败邮件统一进入系统队列，不再依赖个人收件箱和个人记忆力跟踪
- 为售后专员提供最小可用的失败队列查看与处理能力
- 为监督人提供短窗口聚合提醒和固定频率汇总提醒
- 为本周试点建立可验证的成功标准：A 类客户关键失败邮件 100% 进入统一列表/队列

## 三、用户故事
- 作为负责 A 类客户的售后专员，我希望关键批量任务失败邮件被系统统一收进待处理队列，从而不再依赖个人收件箱盯遗漏。
- 作为售后主管或监督人，我希望系统在发现关键失败邮件后给我及时提醒和周期汇总，从而尽早发现未处理风险。
- 作为管理员，我希望可以配置 A 类客户范围和失败邮件识别规则，从而让试点邮箱的捕获范围可控、可调整。

## 四、主要流程
1. 管理员配置试点邮箱范围、A 类客户名单和失败邮件识别规则。
2. 系统按固定频率自动轮询试点邮箱，也支持人工手动补跑。
3. 系统读取邮件后，先做规则匹配，识别是否属于 A 类客户关键批量任务失败邮件。
4. 命中后，系统提取最小必要字段，例如客户信息、任务标识、主题、发件人、接收时间和命中规则快照。
5. 系统依据唯一约束做幂等防重，确保同一封失败邮件不会重复进入队列。
6. 新命中的失败邮件进入失败邮件队列，初始状态为 `new`。
7. 系统在短窗口内聚合新增失败邮件并向监督人发送提醒，同时按固定频率发送汇总邮件。
8. 售后专员登录系统查看失败队列和详情，并将状态从 `new` 推进到 `acknowledged` 或 `resolved`。

## 五、边界情况
- 试点只覆盖明确的一组现有邮箱，不承诺全部门所有邮箱都被纳入。
- 可以新增一个收件箱作为补强入口，但不能依赖所有客户都切换到新收件箱。
- 第一版优先使用规则识别，不以 AI 判定作为主链路成立条件。
- 同一封邮件被自动轮询和手动补跑重复扫描时，必须避免重复入队。
- 规则匹配成功但任务标识缺失时，仍允许入队，但要保留“字段缺失”状态供人工判断。
- 某个邮箱拉取失败时，不能影响其他试点邮箱继续扫描。
- 提醒邮件发送失败时，不能影响失败邮件入队和状态维护。
- 第一版只做最小登录和 `admin/operator` 两级角色，不扩展复杂权限模型。
- 第一版不做完整共享邮箱协作、不做完整 AI 自动处理闭环、不做全量邮件归档平台整合。

## 六、前端改动范围
- 新增最小登录页
- 新增失败邮件队列页
- 新增失败邮件详情页
- 支持状态流转操作：`new -> acknowledged -> resolved`
- 展示命中规则快照、基础提取字段、提醒与处理状态
- 处理登录失败、空队列、接口异常、状态冲突等基础交互
- 与契约字段保持完全一致，不把失败队列硬塞进现有归档页

## 七、后端改动范围
- 新增最小登录接口和角色信息返回
- 新增试点邮箱范围控制能力
- 新增 A 类客户和失败识别规则的最小配置接口
- 新增独立失败邮件队列表与状态流转接口
- 新增手动补跑接口
- 新增后台轮询任务、幂等防重和短窗口提醒逻辑
- 复用现有发信能力输出即时提醒与固定频率汇总
- 增加任务记录、命中快照、错误日志和审计字段

## 八、验收标准
- [ ] 在明确的一组试点邮箱范围内，A 类客户关键失败邮件可 100% 进入统一列表/队列
- [ ] 自动轮询和手动补跑都可触发失败邮件捕获，且不会产生重复队列项
- [ ] 售后专员可登录系统查看失败队列和详情
- [ ] 售后专员可将状态从 `new` 变更为 `acknowledged`，再变更为 `resolved`
- [ ] 监督人可收到短窗口聚合提醒邮件
- [ ] 监督人可收到固定频率汇总邮件
- [ ] 最小登录与 `admin/operator` 两级角色可正常工作
- [ ] 试点邮箱范围外的数据不会被非授权用户看到
- [ ] 提醒发送失败不会影响失败邮件入队
- [ ] 真实失败邮件样本可固化为回放回归测试基线

## 九、测试点
- 正常命中 A 类客户关键失败邮件并成功入队
- 非目标邮件不应误入队
- 同一封邮件重复扫描的幂等防重
- 自动轮询与手动补跑并发时的任务锁与防重
- 规则禁用、规则优先级变化、规则修改后的匹配行为
- 任务标识缺失、正文编码异常、字段提取不完整时的降级行为
- operator 登录、查看队列、查看详情、状态流转
- admin 查看/维护规则配置
- 试点邮箱范围控制与角色权限控制
- 短窗口聚合提醒、固定频率汇总邮件发送
- 邮箱拉取失败、邮件发送失败、状态冲突、资源不存在等异常场景
- 真实失败邮件样本回放回归测试

## 十、风险点
- 如果试点邮箱范围定义不清，“100% 收住”将无法验证
- 如果规则定义过宽，会引入误报，降低专员与监督人的信任
- 如果规则定义过窄，会继续漏掉关键失败邮件，直接破坏试点目标
- 自动轮询与手动补跑并发时，若没有任务锁和唯一约束，容易产生重复项
- 最小登录若没有邮箱范围控制，可能导致试点数据越权
- 短窗口聚合提醒如果边界定义不清，可能造成重复提醒或提醒风暴

## 十一、待确认问题
- A 类客户名单由谁维护，更新频率是什么
- “关键批量任务失败邮件”的初始规则集由谁确认，是否有 20 至 50 封历史样本可直接整理
- 监督人具体是谁，短窗口提醒和固定汇总的频率分别是多少
- 试点邮箱范围最终包含哪些邮箱，是否需要按邮箱区分 operator 可见范围
- 最小登录是复用现有内部账号，还是本项目独立维护试点账号

## 十三、开发顺序建议
1. 先完成试点功能说明与接口契约确认
2. 后端先完成最小登录、规则配置、失败队列表和后台任务骨架
3. 前端同步完成最小登录页与失败队列页骨架
4. 后端完成自动轮询、手动补跑、规则匹配、入队与防重
5. 前端完成详情页与状态流转
6. 后端完成短窗口提醒与固定频率汇总
7. 前后端基于真实失败邮件样本做回放验证与联调

## 十四、交付分界建议
- 后端以 `docs/contracts/a-class-failure-mail-capture.yaml` 为交付边界，对外只承诺契约中的请求和响应结构
- 前端以登录、失败队列、详情、状态流转可用且与契约一致为交付边界
- 规则配置、后台轮询、提醒发信、试点邮箱范围控制由后端负责
- 前端不直接依赖后端内部表结构、规则实现细节或任务执行机制
