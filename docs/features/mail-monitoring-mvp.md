# 功能名称：邮件监控与结构化归档 MVP

## 一、背景
当前项目目标是建设一个统一的邮件监控系统，支持接入多个邮箱，持续拉取邮件内容，识别关键业务信息，并进行结构化归档。同时，系统需要支持按周期发送汇总邮件，便于业务负责人快速查看重要邮件、异常情况和处理结果。

## 二、目标
- 支持多个邮箱接入与独立管理
- 首期仅支持 IMAP 协议接入
- 支持按邮箱拉取邮件并去重入库
- 支持基于 AI 提取的关键内容识别与结构化归档
- 支持归档结果查询、筛选和查看详情
- 支持按日定时或手动发送汇总邮件
- 汇总邮件统一通过 SMTP 发送

## 三、用户故事
- 作为运营或业务负责人，我希望系统可以同时监控多个邮箱，从而统一查看不同邮箱中的关键邮件。
- 作为处理人员，我希望系统能从邮件中识别关键字段并结构化归档，从而减少手工整理成本。
- 作为管理者，我希望按日收到汇总邮件，从而快速了解新增重要邮件、异常邮件和待处理事项。

## 四、主要流程
1. 管理员新增邮箱接入配置，并设置启用状态。
2. 系统按计划任务或手动触发方式，按邮箱拉取新邮件。
3. 系统依据邮件唯一标识和邮箱维度执行幂等去重，保存原始邮件元数据。
4. 系统调用 AI 提取链路识别关键内容，生成结构化归档记录。
5. 用户在前端按邮箱、时间范围、处理状态等条件查询归档结果。
6. 系统基于归档结果按日生成 AI 汇总内容，并通过 SMTP 发送到指定收件人。
7. 用户可查看汇总邮件发送结果和失败原因。

## 五、边界情况
- 同一邮箱重复拉取同一封邮件时，必须避免重复归档。
- 不同邮箱中存在相同主题或相同 Message-ID 时，需以“邮箱 + 邮件唯一标识”作为主要幂等维度。
- 邮箱连接失败、认证失败、网络超时需要记录失败原因并支持重试。
- 邮件正文为空、正文编码异常时，系统应保留原始状态并标记解析异常。
- 首期不解析附件，不提取附件内容，也不做附件归档。
- 汇总邮件发送失败时，不能影响邮件拉取和归档主流程。
- 汇总周期内无数据时，需要明确是否发送“空汇总”。

## 六、前端改动范围
- 邮箱管理页面：列表、新增、编辑、启用/停用、连接测试结果展示
- 归档查询页面：按邮箱、时间范围、状态、规则标签筛选
- 归档详情页面：展示原始邮件信息、识别结果、归档状态
- 汇总邮件配置页面：配置收件人、发送周期、启用状态、汇总范围
- 汇总发送记录页面：展示发送时间、收件人、结果、失败原因
- AI 提取结果展示：展示提取字段、置信提示、提取失败原因
- 统一处理加载态、空态、错误提示和操作结果反馈

## 七、后端改动范围
- 新增邮箱管理接口
- 新增邮件拉取任务接口与任务执行逻辑
- 新增 AI 提取任务链路与结构化结果校验逻辑
- 新增邮件、归档、规则、汇总配置相关数据模型
- 新增归档查询接口和详情接口
- 新增汇总邮件配置、手动发送、发送记录查询接口
- 增加任务日志、失败日志和关键审计字段
- 增加大模型调用封装、Prompt 管理、重试与降级策略

## 八、验收标准
- [ ] 可创建、编辑、查询、启用和停用多个邮箱配置
- [ ] 系统可按邮箱独立拉取邮件，并能避免重复入库
- [ ] 系统可基于 AI 提取关键内容并生成结构化归档记录
- [ ] 前端可按邮箱、时间范围、状态查看归档列表与详情
- [ ] 系统可配置汇总邮件规则并支持手动触发发送
- [ ] 系统按日生成汇总内容，并通过 SMTP 发送给指定收件人
- [ ] 系统可记录汇总邮件发送结果，并可查询成功/失败状态
- [ ] 邮箱拉取失败和汇总发送失败均有明确日志与错误信息

## 九、测试点
- 正常创建多个邮箱并独立拉取邮件
- 同一邮箱重复拉取同一封邮件的幂等校验
- 不同邮箱存在相同主题邮件时的归档隔离
- 邮件正文缺失、编码异常时的解析行为
- AI 提取成功、超时、格式不合法、结果缺字段场景
- 归档列表筛选、分页、详情展示
- 汇总邮件手动发送成功
- 汇总邮件定时发送成功
- 汇总邮件无数据时的发送策略
- SMTP 配置错误、收件人错误、发送超时等失败场景
- 权限不足、参数错误、资源不存在等异常场景

## 十、风险点
- 邮件来源复杂，Message-ID、发件时间、正文编码等字段在不同邮件服务商中可能不稳定
- 多邮箱并发拉取可能带来重复处理、锁竞争和任务堆积问题
- 汇总邮件内容过长、收件人过多时，可能触发 SMTP 限制或发送失败
- 大模型提取结果可能出现格式漂移、字段缺失、幻觉和成本波动
- HTML 正文清洗、时区转换和 Prompt 稳定性都可能引入额外复杂度

## 十一、待确认问题
- 大模型调用方式是兼容 OpenAI 风格接口，还是其他自定义网关
- 需要提取的结构化字段清单是否已固定，例如业务类型、优先级、客户名、金额、风险标签
- AI 提取失败时，是否允许回退为仅保存原始邮件和基础元数据
- 汇总邮件无数据时，默认发送“无新增”汇总，还是直接跳过发送

## 十二、前后端任务拆分
### 后端任务
- 完成 MySQL 表结构、SQLAlchemy 模型和数据库初始化策略
- 完成邮箱配置管理接口，支持新增、编辑、分页查询、启停
- 完成 IMAP 拉取服务，按邮箱拉取邮件并写入原始邮件表
- 完成邮件幂等去重逻辑，避免重复入库和重复归档
- 完成 AI 提取服务，输出结构化字段、摘要、风险标签和待办项
- 完成归档查询接口和详情接口
- 完成汇总配置、手动发送、发送记录查询接口
- 完成 SMTP 发信封装、任务日志和失败重试策略

### 前端任务
- 完成应用基础骨架、路由和 API 请求封装
- 完成邮箱管理页面和表单校验
- 完成归档列表页面，支持分页、筛选、状态展示
- 完成归档详情页面，展示原始邮件与 AI 提取结果
- 完成汇总配置页面，支持收件人、时间、邮箱范围配置
- 完成汇总发送记录页面，支持状态筛选和失败原因展示
- 完成通用加载态、空态、错误提示和权限处理

### 联调任务
- 联调邮箱配置接口字段和错误码
- 联调归档列表、详情接口与前端展示映射
- 联调汇总配置和手动发送接口
- 联调分页结构、时间格式、状态枚举和错误提示文案

## 十三、开发顺序建议
1. 后端先完成邮箱配置接口、数据库表和健康检查接口
2. 前端同步完成应用骨架和邮箱管理页面
3. 后端完成 IMAP 拉取入库和原始邮件持久化
4. 后端完成 AI 提取和归档查询接口
5. 前端完成归档列表和详情页面
6. 后端完成汇总配置、汇总发送和发送记录接口
7. 前端完成汇总配置和发送记录页面
8. 前后端联合测试异常场景、幂等场景和无数据场景

## 十四、交付分界建议
- 后端以接口契约为交付边界，对外只承诺 `docs/contracts` 中定义的请求和响应结构
- 前端以页面可用性和接口契约一致性为交付边界，不依赖后端内部模型命名
- AI 提取链路、IMAP 拉取链路、SMTP 发信链路统一由后端负责
- 前端不直接持有大模型 API Key、SMTP 凭据或 IMAP 凭据
