# 功能名称：原始邮件可见性

## 一、背景
当前邮箱拉取已经可以把邮件写入 `mail_messages`，但前端可见页面只有失败邮件队列和归档列表。用户在邮箱管理里点击“拉取”后，即使邮件已经成功入库，也无法在页面上直接看到结果，容易误判为“拉取失败”。

## 二、目标
提供一个独立的“原始邮件”可见层，让用户可以直接查看某个邮箱最新拉取到的邮件列表和详情，先解决“拉到了但看不见”的问题，再决定后续如何流转到归档或失败识别。

## 三、用户故事
作为 邮箱运营或试点用户，我希望 在系统里直接查看某个邮箱最近拉取到的原始邮件，从而 快速确认拉取是否成功，并判断下一步是否需要进入归档或失败识别。

## 四、主要流程
1. 用户在“邮箱管理”页点击某个邮箱的“查看原始邮件”。
2. 系统跳转到“原始邮件列表”页，并自动按当前邮箱筛选。
3. 用户查看主题、发件人、接收时间、解析状态、提取状态等基础信息。
4. 用户点击某条记录进入详情页，查看正文、发件人、收件人、原始标识等信息。

## 五、边界情况
- 邮箱已拉取成功但暂无邮件：列表显示空态。
- 邮件正文只有 `body_text` 或只有 `body_html`：详情页分别兜底展示。
- `received_at`、`internet_message_id`、`provider_uid` 为空：页面显示 `-`，不报错。
- 详情记录不存在：展示明确空态，不永久 loading。

## 六、前端改动范围
- 新增原始邮件列表页。
- 新增原始邮件详情页。
- 新增原始邮件 API 调用与类型定义。
- 在左侧导航增加“原始邮件”入口。
- 在邮箱管理页增加“查看原始邮件”快捷入口。
- 处理空态、异常态、正文 HTML 安全渲染。

## 七、后端改动范围
- 新增原始邮件列表接口。
- 新增原始邮件详情接口。
- 复用现有 `mail_messages` 数据模型，不做数据库变更。
- 列表支持最小筛选：`mailbox_id`、`keyword`、分页。
- 详情返回正文和基础头信息。

## 八、验收标准
- [ ] 拉取成功后的邮件能在“原始邮件列表”页直接看到。
- [ ] 用户可以按邮箱过滤原始邮件列表。
- [ ] 用户可以进入原始邮件详情查看正文和基础元数据。
- [ ] 空时间、空正文、空标识不会导致页面报错。
- [ ] 邮箱管理页可以直接跳转到当前邮箱的原始邮件列表。

## 九、测试点
- 正常查看原始邮件列表。
- 按邮箱筛选原始邮件。
- 关键词搜索主题或发件人。
- 查看原始邮件详情。
- 空数据空态展示。
- 详情不存在时错误态展示。

## 十、风险点
- 原始邮件列表展示的是 `mail_messages`，不代表已经完成 AI 归档或失败识别，用户可能需要文案区分。
- 邮件 HTML 正文需要消毒渲染，避免 XSS 风险。

## 十一、待确认问题
- 后续是否需要从原始邮件详情直接触发“归档/失败识别”动作。
- 原始邮件列表是否需要增加更多筛选条件，如 `extraction_status`、时间范围。
