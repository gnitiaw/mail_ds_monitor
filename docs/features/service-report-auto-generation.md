# 功能名称：巡检与运维服务报告自动生成

## 一、背景
当前系统已经具备邮件拉取、结构化归档、分析汇总等能力，但围绕客户交付的“巡检报告/运维服务报告”仍缺少一条稳定主链路。实际工作中，交付人员通常需要从服务器巡检结果、漏洞修复记录、运维工作记录、禅道缺陷清单等多个来源手工汇总，再按月报、季报或年报模板整理成正式报告。

这个过程的主要问题不是写作本身，而是：

1. 多源数据分散，时间窗口筛选和统计口径不一致。
2. 数字、典型事项、整改进展需要反复人工核对。
3. 月报、季报、年报往往重复劳动，只是统计周期不同。
4. 报告生成过程不可追溯，后续很难解释某个数字来自哪里。

这个功能的目标不是做一个通用 BI 平台，也不是把现有邮件监控系统的全量记录再搬进一套报告库，而是在现有系统中新增一个“服务报告生成层”，把报告真正需要的运维证据统一归一，并基于模板生成可预览、可导出、可复核的服务报告。

## 二、目标
- 让用户可以为某个项目或客户配置服务报告生成规则，而不是每次手工拼装。
- 让系统自动汇总巡检、漏洞、工作记录、禅道缺陷四类数据，形成统一报告快照。
- 让月报、季报、年报共享同一套生成链路和前端页面，但各自使用独立章节结构与模板。
- 让报告内容中的关键数字可追溯到本次生成使用的数据快照。
- 让用户先预览草稿，再导出正式报告，降低错误外发风险。
- 明确报告域只保存快照、统计、章节草稿和证据引用，不承担原始监控明细沉淀职责。
- 为每份报告输出 `completeness_status`，明确它是“可外发”“仅供内部复核”还是“禁止导出”。
- 明确试点项目的 `project_owner`、模板口径的 `template_owner`、指标口径的 `metric_owner`，避免口径漂移。

## 三、用户故事
- 作为运维服务负责人，我希望为某个客户配置一套服务报告规则，让系统在每个统计周期自动汇总数据并生成草稿，从而减少每次手工整理报告的时间。
- 作为交付经理，我希望在生成报告前看到本次汇总使用的数据快照、统计结果和风险摘要，从而先完成复核，再决定是否导出。
- 作为管理者，我希望月报、季报和年报共享统一口径，避免不同人手工整理时出现数字不一致或章节缺失。
- 作为报告复核人，我希望系统明确告诉我这份报告是 `ready`、`partial` 还是 `blocked`，从而避免把一份看起来完整但数据不全的报告发给客户。

## 四、主要流程
1. `admin` 创建服务报告配置，填写项目名称、报告类型、时间窗口规则、模板、接收对象和 owner 信息。
2. 用户为配置绑定数据源，至少包含：
   - 巡检结果
   - 漏洞修复记录
   - 运维工作记录
   - 禅道缺陷清单
3. 用户触发一次数据汇总，系统在指定时间窗口内拉取或导入数据，并归一为统一报告快照。
4. 系统基于报告快照计算统计项、聚合典型事项，并为整份报告和每个章节计算状态：
   - `completeness_status`
   - `data_status`
5. 系统生成章节化草稿内容。
6. 用户在页面查看报告草稿、源数据概览、关键指标和可信度状态。
7. 当 `completeness_status=ready` 时，用户可以作为可外发报告导出；当 `partial` 时，只允许内部复核使用；当 `blocked` 时，禁止导出。
8. 用户确认后执行导出，系统生成 markdown 或 html 报告文件，并保存本次生成记录。
9. 后续季报和年报复用相同流程，仅更换时间窗口和模板。
10. 如果未来引用原有邮件监控结果，只按报告规则提取统计指标或典型事项，不把全量邮件监控记录写入报告记录。

## 五、边界情况
- 当某个数据源在时间窗口内无数据时，不能直接生成空白章节，必须明确展示“本周期无数据”或按模板规则隐藏章节。
- 当部分数据源接入失败时，系统必须标记哪些章节基于不完整数据生成，不能静默忽略。
- 报告生成所用的数字必须来自结构化汇总结果，AI 只允许做摘要、语言润色和风险归纳。
- 同一配置、同一报告类型、同一时间窗口重复生成时，应优先复用已有活动 `report_run` 或明确阻止重复提交。
- 月报、季报、年报共用运行链路，但章节结构独立，且必须保留模板版本号，避免后续模板变化导致历史报告内容漂移。
- 第一阶段不要求所有外部系统都实时 API 集成，允许“自动拉取 + 标准化导入”并存。
- 第一阶段不输出复杂审批流，先保证“生成、预览、导出”链路成立。
- 原有邮件监控系统若被复用，只允许进入报告快照级别的统计和证据引用，不允许把全量原始邮件、归档或告警记录复制到 `service_report_runs`。
- `completeness_status=blocked` 的报告禁止导出。
- `completeness_status=partial` 的报告可以预览，但必须明确标记“仅供内部复核，不建议直接外发”。
- V1 只支持单项目试点：
  - 所有启用中的配置必须属于同一个 `project_name`
  - 若创建第二个启用中的不同项目配置，应返回冲突

- 新增独立报告域模型，避免污染现有 `summary_configs` 语义：
  - `service_report_configs`
  - `service_report_source_runs`
  - `service_report_runs`
- 新增模板枚举和数据源类型枚举。
- 新增服务报告配置 CRUD 接口。
- 新增按配置执行数据汇总接口，输出统一的报告快照。
- 新增报告生成接口，基于快照生成结构化草稿。
- 新增报告详情与列表接口。
- 新增导出接口，首期支持 `markdown`、`html`。
- 服务层新增：
  - 多源数据归一服务
  - 报告指标聚合服务
  - 模板渲染服务
  - AI 摘要增强服务
  - 报告导出服务
  - 报告可信度计算服务
- 数据落库原则：
  - `service_report_runs` 保存 `config_snapshot`
  - 保存 `source_snapshot_summary`
  - 保存 `report_payload`
  - 最多保存少量 `evidence_refs`
  - 不保存原始来源系统全量明细
- 配置落库原则：
  - 每条配置必须明确记录：
    - `project_owner`
    - `template_owner`
    - `metric_owner`
- 日志与审计：
  - 记录谁创建配置、谁触发汇总、谁触发生成、谁导出
- 权限控制：
  - `admin` 可创建和修改配置
  - `operator` 可执行汇总、生成、查看和导出
  - V1 不引入独立 `viewer` 角色承诺

## 八、验收标准
- [ ] 用户可以创建服务报告配置并选择报告类型（月报、季报、年报）
- [ ] 用户可以为配置绑定四类数据源
- [ ] 系统可以按指定时间窗口执行数据汇总并返回统一快照
- [ ] 系统可以基于快照生成一份章节化草稿报告
- [ ] 报告详情页可以展示各数据源同步状态和关键统计指标
- [ ] 报告中的核心数字可以追溯到本次生成快照
- [ ] 每份报告都有 `completeness_status`
- [ ] 每个章节都有 `data_status`
- [ ] `blocked` 报告无法导出
- [ ] `partial` 报告会明确提示“仅供内部复核”
- [ ] 用户可以导出 markdown 或 html 格式的报告
- [ ] 同一时间窗口重复触发生成时有明确幂等策略
- [ ] 某个数据源缺失或失败时，页面有明确提示，不会静默忽略
- [ ] 前端页面共用同一套配置、运行列表和详情页
- [ ] 月报、季报、年报按各自独立章节结构成功生成
- [ ] 报告运行记录中不复制原系统全量明细，只保存报告所需快照和证据引用
- [ ] 启用中的配置只能属于同一个试点项目
- [ ] 每条配置都有明确的 `project_owner`、`template_owner`、`metric_owner`

## 九、测试点
- 正常流程：
  - 创建报告配置成功
  - 执行数据汇总成功
  - 生成报告草稿成功
  - 报告导出成功
- 参数错误：
  - `report_type` 非法
  - 时间窗口格式错误
  - `template_key` 不存在
  - `source_bindings` 缺少必需数据源
  - owner 字段缺失
- 权限不足：
  - 未登录访问接口
  - `operator` 尝试创建或修改配置
  - 非 `admin/operator` 用户访问服务报告接口
- 状态冲突：
  - 同一配置同一窗口重复触发汇总
  - 数据汇总未完成时触发报告生成
  - 失败状态的 `report_run` 直接导出
  - 第二个不同 `project_name` 的启用中配置创建失败
- 数据边界：
  - 报告明细页不返回原始全量邮件监控记录
  - 报告记录只返回摘要、统计和有限证据引用
- 空数据：
  - 某个数据源没有记录
  - 四类数据源全部为空
  - 只有工作记录，其他统计为空
- 部分失败：
  - 巡检数据成功，禅道数据失败
  - 漏洞数据缺字段导致归一失败
- 重复提交：
  - 短时间内重复点击生成
  - 已存在相同时间窗口成功报告时再次触发

## 十、风险点
- 真实难点在数据口径统一，不在页面展示。若不先固化快照结构，后续所有章节都会漂。
- 外部数据源接口能力未知，如果强推全量自动对接，进度风险很高。
- 运维工作记录可能高度非结构化，若没有最小字段规范，报告只能生成很弱的摘要。
- 如果报告模板不固定，第一版会很快失控，前后端都难以收敛。
- 导出格式若一开始就做 PDF/Word 双支持，工作量会明显膨胀。
- 若不保存模板版本和数据快照，历史报告无法审计，也不利于复盘。
- 如果不提前划清“原系统主数据”和“报告快照”的边界，后面很容易把报告模块做成冗余总库，数据同步和一致性成本会迅速失控。
- 如果不把 `template_owner`、`metric_owner`、`project_owner` 提前固定，系统只会更快地把口径分歧固化。
- 如果不引入 `completeness_status`，系统很容易生成一份“看起来完整、其实不适合外发”的报告。

## 十二、首期范围表
| 范围项 | 首期结论 | 在报告中的落地方式 | 说明 |
| --- | --- | --- | --- |
| 巡检结果 | 必须进入 | 统计摘要 + 章节内容 | 这是服务报告主口径之一，首期必须支持 |
| 漏洞修复记录 | 必须进入 | 统计摘要 + 章节内容 | 直接影响客户对安全服务价值的感知 |
| 运维工作记录 | 必须进入 | 统计摘要 + 典型事项摘要 | 允许来源较弱，但至少要形成工作量和重点事项口径 |
| 禅道缺陷清单 | 必须进入 | 统计摘要 + 章节内容 | 首期以项目维度缺陷统计和处理结果为主 |
| 单项目试点 | 必须遵守 | 启用中配置只能属于同一个 `project_name` | 这是 V1 硬约束，不在本轮放开 |
| 原系统邮件监控结果 | 默认不进入主口径 | 暂不纳入首期报告主数据源 | 原系统继续做主存储和明细工作台，报告模块不复制全量数据 |
| 邮件监控统计指标 | 可选预留 | 仅允许汇总指标 | 例如本周期告警数、关闭数，只有在报告模板明确需要时才接入 |
| 邮件监控典型事件 | 可选预留 | 仅允许 `evidence_refs` 引用 | 只保留少量案例引用，不把原始邮件、原始归档全文写入报告记录 |
| 原系统全量邮件明细/归档明细/告警明细 | 明确不进入 | 不落入 `service_report_runs` | 这是边界红线，避免报告模块变成第二套总库 |
| PDF / DOCX 导出 | 首期不做 | 暂不支持 | 首期先用 `markdown/html` 跑通链路 |
| 自动审批 / 复杂流转 | 首期不做 | 暂不支持 | 先保证“生成、预览、导出”闭环 |

首期建议就按这张表执行，不再继续扩口径。尤其是“原系统邮件监控结果”这一项，先把它定义成报告域外能力，最多后续按模板需要做统计引用，不要反过来把报告系统做成监控数据副本。

## 十四、测试计划
1. 先用单个项目、单个月报模板做样本验证
2. 用真实样本验证四类数据源是否能映射到统一快照结构
3. 验证同一窗口重复触发是否符合幂等预期
4. 验证部分数据源失败时，报告详情提示是否清晰
5. 验证导出文件内容与页面预览一致
6. 验证季报、年报仅切换时间窗口后能否复用同一链路
7. 验证报告详情只展示快照、统计和证据引用，不泄露原系统全量明细
8. 验证 `ready / partial / blocked` 三种状态是否符合规则
9. 验证第二个不同项目的启用配置会被拦截
