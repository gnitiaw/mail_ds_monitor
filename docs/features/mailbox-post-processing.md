# 功能名称：邮箱已拉取邮件处理

## 一、背景
当前系统已经可以把邮件从 IMAP 拉取到 `mail_messages`，也已经具备两条后续能力：
- AI 提取并归档到 `archive_records`
- 按失败规则识别并进入 `failure_queue`

但这两条能力没有和邮箱拉取后的日常操作形成明确入口，导致用户即使看到了原始邮件，仍然需要猜系统是否已经完成后处理。

## 二、目标
为邮箱提供一个明确的“处理已拉取邮件”动作，把当前邮箱中已拉取但未处理的邮件送入归档与失败识别链路，让用户能从邮箱管理页直接完成“拉取 -> 处理 -> 查看结果”。

## 三、用户故事
作为 运营或试点用户，我希望 在拉取邮箱后手动触发后处理，从而 让原始邮件进入归档列表和失败邮件队列，验证整条业务链路。

## 四、主要流程
1. 用户在邮箱管理页点击“处理已拉取邮件”。
2. 系统按邮箱范围处理该邮箱内的待提取邮件。
3. 系统对该邮箱最近时间窗口内的邮件执行失败规则匹配。
4. 系统返回处理统计结果。
5. 用户可以去“归档列表”或“失败邮件队列”查看产出。

## 五、边界情况
- 邮箱存在但当前没有待处理邮件：返回 0 条处理结果。
- LLM 未配置或关闭：归档仍可创建，但提取状态可能失败或记录降级原因。
- 没有启用的失败规则：失败队列处理结果为 0，不报错。
- 重复处理同一邮箱：归档按 message_id 防重，失败队列按既有幂等规则防重。

## 六、前端改动范围
- 邮箱管理页增加“处理已拉取邮件”按钮。
- 调用新接口并提示处理结果。
- 不新增复杂工作台，只提供最小操作入口。

## 七、后端改动范围
- 新增邮箱后处理接口。
- 扩展提取服务，支持按邮箱范围处理待提取邮件。
- 复用失败识别服务，支持扫描已有邮件而不重复拉取。
- 返回最小统计结果，便于前端提示。

## 八、验收标准
- [ ] 用户可以在邮箱管理页手动触发“处理已拉取邮件”。
- [ ] 接口返回归档处理数量和失败识别数量。
- [ ] 处理后，原始邮件可以出现在归档列表或失败邮件队列中。
- [ ] 重复处理同一批邮件不会无限重复入队。

## 九、测试点
- 正常处理含待提取邮件的邮箱。
- 无待处理邮件时返回 0。
- 失败规则命中时进入失败队列。
- 重复处理时幂等。
- 邮箱不存在时返回 404。

## 十、风险点
- 如果 LLM 配置不可用，归档链路可能失败，但失败识别链路仍可工作。
- 当前接口是同步处理，邮箱邮件很多时会拉长请求时间。

## 十一、待确认问题
- 后续是否要把该动作并入“拉取”接口，作为默认副作用。
- 是否需要把处理结果明细化到“归档成功/归档失败/失败队列命中/失败队列去重”。
