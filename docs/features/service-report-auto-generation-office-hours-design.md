## 问题陈述
当前巡检报告、月度运维服务报告、季度服务报告和年度服务报告，往往还是靠人从多个来源手工拼出来：

1. 服务器巡检结果
2. 漏洞修复记录
3. 运维工作记录
4. 禅道缺陷清单
5. 其他补充说明材料

真正费时间的不是“把一段话写漂亮”，而是把不同口径的数据拉齐、筛到同一时间窗口、确认统计口径没错、再套进固定模板。这个过程重复、慢、容易漏，也很难保证每个月写出来的内容口径一致。

所以这个功能第一阶段要解决的问题，不是“做一个万能 BI 平台”，也不是“让 AI 自己编一份报告”，而是先把服务报告生成链路做成稳定的：

**按固定时间范围，把多源运维证据统一汇总成一个结构化快照，再基于模板自动生成可预览、可导出、可复核的服务报告。**

## 需求证据
- 报告内容高度重复，天然适合模板化和结构化。
- 现有流程的主要成本是“数据搜集和口径校对”，不是排版本身。
- 月报、季报、年报本质只是时间窗口不同，底层生成链路应该共用。
- 这类报告通常面向客户或管理层，口径不一致、缺项、数字对不上，会直接伤害交付可信度。
- 当前仓库已经有 `summary / analysis / send` 这类“按时间窗口汇总并生成内容”的能力基础，可以复用思路，但不应硬塞进旧邮件汇总对象里。

## 现状工作流
1. 运维人员分别从巡检表、漏洞台账、工单记录、禅道列表里导出数据。
2. 人工筛选本月、本季度或本年的时间窗口。
3. 人工汇总核心数字，比如巡检次数、漏洞处理数量、缺陷关闭情况、运维工作量。
4. 人工挑选典型事项，写进报告正文。
5. 人工套 Word 或内部模板。
6. 复核时再次发现数字、时间范围或分类口径不一致，来回改。
7. 最终导出报告并发送。

当前真正的竞争对手不是别家的系统，而是“Excel 导出 + Word 模板 + 人肉核对 + 临交付前加班”。

## 目标用户与最小切口
- **核心用户：** 运维服务负责人、交付负责人、项目经理
- **第一阶段最小切口：** 单项目服务报告生成
- **首期报告类型：** 月报、季报、年报都支持，前端页面复用同一链路，但章节结构独立
- **首期支持数据源：**
  - 巡检结果
  - 漏洞修复记录
  - 运维工作记录
  - 禅道缺陷清单
- **首期成功标准：** 用户可以为一个项目配置报告模板和数据源绑定，系统在指定时间窗口内生成一份结构化草稿报告，并支持人工复核后导出

## 约束
- 当前系统主体是邮件监控与汇总，不是现成的运维数据平台。
- 原有邮件监控系统仍然是邮件类数据的主存储和主工作台，不应该为了做服务报告，把全量邮件监控记录再复制进报告域。
- 外部数据源未必都有稳定 API，也未必当前就具备统一鉴权方式。
- 禅道、漏洞平台、巡检工具、运维记录来源很可能口径不一致。
- 第一阶段如果一开始就做“全自动外部系统全集成”，项目会明显变重。
- 对外输出的服务报告不能只依赖 LLM 自由发挥，关键数字必须可追溯。

## 前提
1. 报告的核心资产是“结构化事实快照”，不是最终文案。
2. 先把数据口径和模板稳定下来，再谈高级排版、智能润色和更多系统接入。
3. 月报、季报、年报不该做三套系统，而应该共享同一套生成与导出链路。
4. 报告域只保存报告所需的统计快照、章节草稿和少量证据引用，不保存各来源系统的全量原始记录。
5. 首期必须明确 `project_owner`、`template_owner`、`metric_owner`，否则系统只会更快地把口径分歧固化。
6. 首期只允许单项目试点，不做多项目并行扩展。

## 方案对比

### 方案 A：服务报告生成层
**摘要**
新增独立的服务报告模块，维护报告配置、模板、数据源绑定、汇总快照和报告生成记录。首期支持按配置拉取或导入数据，统一归一后生成报告草稿。

**投入**
- Effort: M
- Risk: Low

**优点**
- 最贴合当前痛点，先解决“拼报告太慢、太散、容易错”
- 月报、季报、年报共用一套能力
- 数据快照、报告草稿、导出结果可以追溯
- 不会把当前 `summary_configs` 语义搞乱
- 也不会把现有邮件监控系统变成“什么都往报告里塞”的第二套总库

**缺点**
- 需要新增一套报告对象模型
- 首期外部数据源接入不可能一步到位
- 模板和指标口径需要先达成一致

**复用**
- 现有时间窗口汇总思路
- 现有 AI 摘要与发送链路中的部分内容生成能力
- 现有权限模型以 `admin / operator` 为准，V1 不新增 `viewer`

### 方案 B：直接扩展现有 Summary 模块
**摘要**
把服务报告也塞进现有 `summary_configs / summary_sends / analysis_runs` 体系，新增几个模式字段后复用全部页面和接口。

**投入**
- Effort: S
- Risk: High

**优点**
- 看起来复用更多
- 页面和接口改动面可能更小

**缺点**
- 旧汇总主链路本来面向邮件摘要，语义会迅速变脏
- 前后端会出现“到底是邮件汇总还是服务报告”的双真相
- 报告草稿、模板、导出、数据源绑定这些概念会硬塞进不合适的对象

**复用**
- 仅复用表面接口和页面壳子

### 方案 C：先只做模板写作助手
**摘要**
不做数据统一，只让用户把各种材料贴进来，系统用 LLM 按模板生成一份报告文案。

**投入**
- Effort: S
- Risk: Med

**优点**
- 上线快
- 演示效果容易做

**缺点**
- 不能真正解决数据汇总和口径统一问题
- 数字不可追溯，风险很高
- 复核成本依然在，甚至可能更高

**复用**
- 现有 LLM 文本生成能力

## 推荐方案
推荐 **方案 A：服务报告生成层**。

这事最值钱的地方，不是“把话写得更像报告”，而是把多源事实统一成一个可复核的报告快照。方案 A 正好卡在这个价值点上。它解决的不是写作表面，而是交付链路本身。

## 推荐落地形态
推荐把第一版定义成一个“报告配置 + 数据快照 + 草稿生成 + 导出”的闭环，而不是一个大而全运维平台。

### 第一版必须有的能力
1. 配置报告对象：
   - 项目/客户
   - 报告类型（月报、季报、年报）
   - 时间窗口规则
   - 模板
   - 数据源绑定
2. 统一汇总四类数据源：
   - 巡检结果
   - 漏洞修复记录
   - 运维工作记录
   - 禅道缺陷清单
3. 生成一份结构化报告草稿，包含：
   - 概览摘要
   - 巡检统计
   - 漏洞修复情况
   - 运维工作记录摘要
   - 缺陷处理情况
   - 风险与下阶段建议
4. 报告草稿支持页面预览
5. 报告支持导出为 markdown/html，PDF 放到第二阶段
6. 保存本次生成所用的时间窗口、数据快照和模板版本
7. 若未来需要引用邮件监控结果，只保存报告章节需要的统计摘要和证据引用，不复制全量邮件明细
8. 为每份报告输出 `completeness_status`，明确它是：
   - 可外发
   - 仅供内部复核
   - 禁止导出

### 第一版不应该有的能力
1. 不做通用 BI 看板
2. 不做所有外部平台的一次性深度双向集成
3. 不把所有自由文本内容都交给 AI 自动生成
4. 不在第一版引入复杂审批流
5. 不承诺所有数据源都实时同步
6. 不把原系统里的全量邮件监控、告警明细、归档明细整体搬进报告记录

## 关键设计决定

### 1. 先归一事实，再生成文案
关键数字和表格必须来自结构化汇总结果。AI 只负责摘要润色、风险归纳、语句组织，不负责发明事实。

### 2. 前端页面共用，但三种周期章节结构独立
季报和年报只是时间窗口更长，但客户阅读重点不同，所以不需要做三套系统，却需要三套章节结构和模板。

### 3. 首期允许“自动拉取 + 手工导入”并存
如果某些数据源暂时没有稳定 API，就允许先用标准化导入。目标是先打通报告生成，不是被外部系统接入拖死。

### 4. 生成对象要保存快照
报告一旦生成，就要保存当时的源数据统计和模板版本。否则后面再看同一个月报，数字会漂。

### 5. 草稿和导出分离
先生成可预览草稿，再做导出。这样用户能先校对，不会直接把一份错误 PDF 发出去。

### 6. 报告只保留“引用”，不保留“全量明细”
这点很重要。原系统里的邮件监控、告警归档、明细查询，继续留在原系统对象里。报告模块只拿最终要进报告的内容，比如：

- 本周期告警总数
- 已关闭/未关闭数量
- 典型事件摘要
- 少量可追溯的证据引用 ID

不要把所有原始邮件、原始告警、原始归档明细再复制到 `report_run` 里。那样很快就会把报告系统做成一个低配数据仓库。没必要。

### 7. 报告必须自带可信度状态
系统不能只会生成一份“看起来像报告”的文档，还得告诉用户这份东西到底能不能发。

最少要有三种状态：
- `ready`
- `partial`
- `blocked`

规则也要很简单：
- `ready` 可以外发
- `partial` 只建议内部复核
- `blocked` 禁止导出

没有这层，系统很容易生成一份漂亮但不可信的报告。

### 8. 必须把 owner 钉死
这类系统最怕的不是没数据，而是口径漂。

V1 至少要固定：
- `project_owner`
- `template_owner`
- `metric_owner`

否则后面每个人都会觉得“我理解的月报是对的”，系统会更快把分歧产品化。

## 开放问题
1. 首期外部数据源到底有哪些现成接口可用，哪些只能导入？
2. 月报模板是否已经固定，还是需要系统内维护多个模板版本？
3. 季报和年报是否允许覆盖部分章节，比如季度重点事项、年度趋势分析？
4. 报告最终要导出哪些格式：
   - markdown
   - html
   - pdf
   - docx
5. 运维工作记录的结构化来源是什么，是系统工单、Excel，还是自由文本？
6. 禅道缺陷统计是按项目、产品、模块还是自定义查询维度汇总？
7. 后续如果需要在报告里体现邮件监控结果，范围是：
   - 完全不纳入
   - 只纳入统计指标
   - 纳入统计指标 + 典型问题引用
8. V1 试点项目是谁，`project_owner`、`template_owner`、`metric_owner` 分别由谁担任？

## 成功标准
第一版的成功，不看页面做得多全，而看下面几件事：

1. 用户可以为一个项目创建报告配置
2. 系统可以在指定时间窗口内汇总四类数据源
3. 系统能生成一份结构化草稿报告
4. 报告中的核心统计数字可追溯到数据快照
5. 用户可以预览、复核并导出报告
6. 月报生成链路跑通后，季报和年报无需重做一套系统

## 首期范围表
| 范围项 | 首期结论 | 处理原则 |
| --- | --- | --- |
| 巡检结果 | 必做 | 进入报告主口径 |
| 漏洞修复记录 | 必做 | 进入报告主口径 |
| 运维工作记录 | 必做 | 进入报告主口径 |
| 禅道缺陷清单 | 必做 | 进入报告主口径 |
| 原系统邮件监控全量记录 | 不做 | 保留在原系统，不进入报告域 |
| 邮件监控统计指标 | 预留 | 后续若模板需要，可作为统计引用接入 |
| 邮件监控典型事件 | 预留 | 只允许少量证据引用，不复制全文明细 |
| PDF / DOCX 导出 | 不做 | 第二阶段再补 |

这张表的作用只有一个，防止范围失控。报告系统只做“客户要看到的结果”，不做“把内部所有监控数据再存一遍”。

## 依赖与前置条件
- 需要确定首期目标项目或客户范围
- 需要确定月报模板的固定章节
- 需要明确四类数据源的最小接入方式
- 需要明确最终导出格式优先级
- 需要确认哪些角色可以创建配置、生成草稿、导出报告

## 本周最该完成的一件事
不要先去做页面，也不要先去写 PDF 导出。

**这周最该完成的是先拿一份真实月报模板，把四类数据源字段映射成一个统一的“报告快照结构”，并用最近一个自然月的真实样本手工验证一次。**

如果这一步不做，后面任何“自动生成”都会只是更快地生成一份口径不稳的报告。

## 我注意到你这次需求的方向
- 你盯的是“可交付报告”这个真正会出现在客户面前的产物，不是内部小工具。
- 你把“巡检结果、漏洞修复、工作记录、禅道缺陷”一起提出来，说明你看到问题不在单点系统，而在多源事实无法被稳定整合。
- 这类需求如果范围收不好，很容易变成一个半年都收不住的大平台。现在把第一阶段钉在“服务报告生成层”，是对的。
