# 功能名称：原始邮件提取失败重试

## 一、背景
当前系统已经支持查看原始邮件列表和详情，也能展示 `parse_status`、`extraction_status` 等处理状态。但当邮件在 AI 提取阶段失败后，页面只能看到失败结果，用户无法在系统内直接重试，只能依赖开发或运维手工处理。

同步式“点按钮立刻返回最终结果”的方案在单条场景还勉强可用，但在批量场景会把接口阻塞时间、重复提交、重启恢复、权限边界这些问题一起放大。因此本轮不再把它当成简单按钮补丁，而是补齐一条可恢复的后台任务链路。

## 二、目标
- 允许已登录用户对 `extraction_status=failed` 的原始邮件执行单条或批量重试。
- 单条和批量统一走异步任务模型，接口返回 `job_id`，前端轮询任务状态。
- 通过 `TaskLog` 复用现有任务体系，不新增第三套任务模型。
- 对同一批邮件的重复提交做去重，活跃任务直接复用已有 `job_id`。
- 页面明确展示失败原因、已重试次数和最大重试次数。
- 保持原始邮件列表和详情查询接口向后兼容。

## 三、用户故事
- 作为邮箱运营人员，我希望在原始邮件列表里直接重试失败邮件，并且不用长时间等待接口返回。
- 作为处理人员，我希望在详情页看到失败原因、重试次数和最近一次重试时间，从而判断是否值得继续重试。
- 作为管理员，我希望系统限制最大重试次数，并且对重复点击自动去重，避免模型额度被重复消耗。
- 作为操作员，我只能重试自己邮箱范围内的邮件，不能越权触发别的邮箱任务。

## 四、主要流程
1. 用户进入“原始邮件列表”或“原始邮件详情”。
2. 系统展示邮件的 `extraction_status`、`extraction_error`、`retry_count`、`max_retries`。
3. 当邮件状态为 `failed` 且 `retry_count < max_retries` 时，页面展示“重试”入口。
4. 用户触发单条或批量重试，后端先校验登录身份和 mailbox scope。
5. 后端基于 `message_ids + requested_by + retry_type` 生成稳定 `task_key`。
6. 若已有 `pending/running` 状态的活跃任务，则直接复用已有 `job_id`；否则创建新的 `TaskLog` 并投递到 APScheduler。
7. 接口立即返回 `202 Accepted`，前端进入轮询。
8. worker 批量读取目标邮件，逐条执行重试：
   - 非 `failed` 状态记录为 `not_failed_status`
   - 达到上限记录为 `max_retries_reached`
   - 不存在记录为 `not_found`
   - 可重试记录递增 `retry_count`、更新 `last_retry_at`，并复用现有提取归档链路
9. worker 完成后写回 `TaskLog.result`，前端停止轮询并刷新列表或详情。

## 五、边界情况
- 邮件不存在时：
  - 单条重试返回 `404`
  - 批量重试若全部不存在返回 `404`
  - 批量执行中个别缺失记录记入 `not_found`
- 非 `failed` 状态邮件不能真正重试，worker 仅记录为 `not_failed_status`。
- 单封邮件达到最大重试次数后不能继续重试，worker 记录为 `max_retries_reached`。
- 批量重试单次最多支持 50 条。
- 同一批邮件被重复提交时，不新建第二条活跃任务，直接返回已有 `job_id`。
- 服务重启后，丢失调度器 job 的 `pending/running` 重试任务要在启动阶段标记失败，避免永远停留在 `pending`。
- 本轮只覆盖“提取失败重试”，不覆盖 `parse_status=failed` 的解析失败邮件。

## 六、前端改动范围
- 原始邮件列表页新增：
  - 提取失败原因提示
  - 重试次数列
  - 单条重试按钮
  - 批量选择与批量重试按钮
  - 后台任务轮询中的提示条
- 原始邮件详情页新增：
  - 重试次数与最近重试时间展示
  - 单条重试按钮
  - 后台任务轮询中的提示条
- API 类型定义新增：
  - `max_retries`
  - `TaskLogAcceptedResponse`
  - `TaskLogDetail`
  - 提取重试任务结果结构
- 新增最小前端测试基建：
  - `vitest`
  - `@testing-library/react`
  - MailMessage 列表/详情轮询测试

## 七、后端改动范围
- 扩展原始邮件列表接口返回：
  - `parse_error`
  - `extraction_error`
  - `retry_count`
  - `max_retries`
- 扩展原始邮件详情接口返回：
  - `retry_count`
  - `max_retries`
  - `last_retry_at`
- 单条重试接口改为异步受理，返回 `job_id`
- 批量重试接口改为异步受理，返回 `job_id`
- 新增任务查询接口：
  - `GET /api/v1/task-logs/{job_id}`
- 新增/复用服务能力：
  - 基于 `TaskLog` 的任务创建、去重、状态流转
  - APScheduler 持久化执行与 stuck 任务恢复
  - worker 批量取数、逐条重试、结果汇总
  - `extract_and_archive()` 由外层统一控制事务提交

## 八、验收标准
- [x] 原始邮件列表可展示提取失败原因、重试次数和最大重试次数
- [x] 原始邮件详情可展示重试次数、最大重试次数和最近重试时间
- [x] 单条重试接口返回 `202` 和 `job_id`
- [x] 批量重试接口返回 `202` 和 `job_id`
- [x] 前端会轮询任务状态并在完成后刷新数据
- [x] 重试接口要求登录，并校验 mailbox scope
- [x] 对同一批邮件重复提交会复用已有活跃任务
- [x] 后端会记录 `not_failed_status`、`max_retries_reached`、`not_found` 等批量结果
- [x] 现有原始邮件查看接口保持向后兼容

## 九、测试点
- 后端接口：
  - 单条重试返回 `202` 和 `job_id`
  - 批量重试返回 `202` 和 `job_id`
  - 登录缺失或 Authorization 非法时拒绝访问
  - operator 访问 scope 外邮箱返回 `403`
  - `GET /task-logs/{job_id}` 的 `200/403/404` 分支
  - 同批邮件重复提交复用已有活跃任务
- 后端 worker：
  - 失败归档复用后可重新提取成功
  - 部分失败、部分状态不符、部分超上限、部分不存在时结果统计正确
  - 外层统一提交和失败回滚后，邮件状态与任务结果一致
- 前端：
  - 列表页单条重试后进入轮询，并在任务完成后刷新列表
  - 详情页单条重试后进入轮询，并在任务完成后刷新详情
  - 轮询期间展示任务处理中提示

## 十、风险点
- 当前前端仍依赖轮询而非推送，任务量较大时页面状态刷新存在轻微延迟。
- 批量重试虽然已异步化，但 worker 仍逐条执行提取，吞吐量受模型响应速度影响。
- 现有原始邮件列表和详情读取接口仍未做登录保护，本轮只对写操作和任务查询收口权限。
- `parse_status=failed` 与 `extraction_status=failed` 仍容易被用户混淆，页面文案需要继续打磨。

## 十一、待确认问题
- 是否要在页面显式展示“最多重试 3 次”的规则说明。
- 批量结果是否要在 UI 上单独展示 `not_found` 统计。
- 后续 `parse_status=failed` 的解析重试能力是否复用当前任务模型。

## 十二、前后端任务拆分
### 后端任务
- 扩展 `mail_messages` schema，补充 `max_retries`。
- 将单条和批量重试接口统一改为异步受理。
- 基于 `TaskLog` 补任务创建、去重、状态流转和查询接口。
- 用 APScheduler 持久化执行提取重试任务，并在启动时恢复 stuck 任务。
- 调整 `extract_and_archive()` 事务边界，补充回归测试。

### 前端任务
- 调整重试接口类型为异步 `job_id` 语义。
- 新增任务查询 API。
- 列表页和详情页增加轮询逻辑、处理中提示、结果提示。
- 引入最小测试基建并补 MailMessage 页面测试。

### 联调任务
- 对齐 `TaskLog.result` 的字段口径。
- 对齐单条和批量任务完成后的提示文案。
- 验证 operator 的 mailbox scope 与前端可见行为一致。

## 十三、开发顺序建议
1. 先收口后端契约和任务模型。
2. 完成后端异步执行、鉴权、去重与测试。
3. 再切前端轮询和页面提示。
4. 最后补最小前端测试基建和文档同步。

## 十四、交付分界建议
- 后端以“异步任务模型、状态查询、鉴权、去重、恢复能力”作为交付边界。
- 前端以“列表/详情可触发重试并自动刷新结果”作为交付边界。
- 后续若把轮询改为推送、或把解析失败纳入重试，需要单独走契约变更流程。
