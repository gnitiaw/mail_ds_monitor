# 功能名称：邮件内容 AI 提取智能体规划

## 一、定位
该智能体负责从邮件正文中提取关键业务信息，并输出稳定的结构化结果，供归档、检索和汇总邮件使用。它是后端内部能力，不直接暴露为独立前端页面，但需要提供可追踪的任务状态和结果字段。

## 二、职责范围
- 接收邮件正文、主题、发件人、收件时间等基础信息
- 执行文本清洗和长度裁剪
- 调用外部大模型 API 完成字段提取与摘要生成
- 将模型输出校验为固定 JSON 结构
- 将失败原因、模型信息、耗时、重试结果写入日志

## 三、非目标
- 首期不处理附件内容
- 首期不做多轮 Agent 协作
- 首期不允许前端直接持有模型 API Key

## 四、建议处理链路
1. 邮件拉取后落原始元数据和正文
2. 执行正文清洗，去除冗余 HTML 标签和过长引用内容
3. 生成标准化 Prompt，请求大模型提取结构化字段
4. 校验返回 JSON，补充系统字段，例如 `mailbox_id`、`message_id`
5. 写入归档表和 AI 提取日志表
6. 汇总任务按日读取归档结果，再请求模型生成汇总文案
7. 通过 SMTP 发送汇总邮件

## 五、建议输出结构
```json
{
  "summary": "邮件摘要",
  "business_type": "合同/付款/告警/其他",
  "priority": "high/medium/low",
  "risk_tags": ["逾期", "退款"],
  "action_items": ["今日回访客户"],
  "entities": {
    "customer_name": "示例客户",
    "amount": "10000",
    "currency": "CNY"
  },
  "confidence": 0.92
}
```

## 六、后端模块建议
- `llm_client`：封装大模型 API 调用
- `prompt_templates`：管理提取 Prompt 与汇总 Prompt
- `extractor`：负责请求拼装和结果解析
- `validators`：校验模型输出 JSON 结构
- `archive_mapper`：将提取结果映射到归档字段
- `summary_generator`：生成按日汇总内容

## 七、配置建议
- 大模型 Base URL：仅后端可读
- API Key：仅后端环境变量保存
- Model 名称：可配置
- 请求超时：可配置
- 重试次数：可配置
- Prompt 版本：可配置并留痕

## 八、可靠性策略
- 模型调用失败时，保留原始邮件和基础元数据，不阻塞主链路入库
- 提取结果格式不合法时，进入失败状态并记录原始响应
- 对同一封邮件重复提取时，按 `mailbox_id + message_id` 保持幂等
- 汇总生成失败时，可降级为模板化纯规则汇总

## 九、安全要求
- API Key 不进入前端、不写入仓库、不出现在日志明文中
- 对模型输入做长度控制和敏感信息脱敏预留
- 模型输出入库前做字段白名单校验
